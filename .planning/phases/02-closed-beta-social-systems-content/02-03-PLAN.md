-- ---------- Featured row stalls (12 anchored stands) ---------------------
-- Phase 2 Player Stalls. Each stand is an empty Part tagged with attribute
-- `featuredSlotIndex = N` (1..12). Stall.luau's RentFeaturedSlot handler
-- assigns a renting seller's userId to attribute `stallSellerUserId` on the
-- lowest-indexed unoccupied stand; PromptTriggered routes via that attribute.
-- Hand-built escape valve: if a Part named "FeaturedStall_NN" already exists
-- in Workspace, respect it (just stamp the attribute if missing).
local function buildFeaturedRowStalls()
    local origin = Constants.STALL.featuredRowOrigin
    for i, offset in ipairs(Constants.STALL.featuredRowOffsets) do
        local name = string.format("FeaturedStall_%02d", i)
        local existing = Workspace:FindFirstChild(name)
        if existing and existing:IsA("BasePart") then
            if existing:GetAttribute("featuredSlotIndex") == nil then
                existing:SetAttribute("featuredSlotIndex", i)
            end
        else
            local stall = makePart(
                name,
                Vector3.new(4, 4, 4),
                origin + offset,
                Color3.fromRGB(160, 130, 90),
                Enum.Material.Wood
            )
            stall:SetAttribute("featuredSlotIndex", i)
            stall.Parent = Workspace
            -- Add a SurfaceGui showing "Available" until rented (PLAN 04 polish:
            -- replace with seller name + listing icons via SurfaceGui driven by
            -- replicated state). For now, just a plain text label.
            local sgui = Instance.new("SurfaceGui")
            sgui.Face = Enum.NormalId.Front
            sgui.LightInfluence = 0
            sgui.Parent = stall
            local txt = Instance.new("TextLabel")
            txt.Name = "Label"
            txt.Size = UDim2.fromScale(1, 1)
            txt.BackgroundTransparency = 1
            txt.Text = "Available"
            txt.TextColor3 = Color3.fromRGB(245, 235, 215)
            txt.TextScaled = true
            txt.Font = Enum.Font.GothamBold
            txt.Parent = sgui
            -- ProximityPrompt for buyers to trigger the buyer UI. Stall.start's
            -- PromptTriggered handler reads `stallSellerUserId` (set by RentFeaturedSlot)
            -- to determine if this stall is rented — if not, the prompt is a no-op
            -- (handler returns silently).
            local prompt = Instance.new("ProximityPrompt")
            prompt.Name = "StallPrompt"
            prompt.ActionText = "Browse Stall"
            prompt.ObjectText = "Featured Stall"
            prompt.HoldDuration = 0
            prompt.MaxActivationDistance = 10
            prompt.RequiresLineOfSight = false
            prompt.Parent = stall
        end
    end
end

-- ---------- Plot anchor grid (64 tiles in Cottages zone) ----------------
-- Phase 2 Player Stalls. Each tile is an inert 16x16 Part marking a player's
-- personal stall location. Tagged with attribute `plotAnchorId = "tile_NN"`.
-- On a player's first stall interaction, PlotAnchors.assign claims the lowest
-- unclaimed tile and writes data.stalls.plotAnchorId. Stall.start's
-- PromptTriggered routes prompt → OpenStallBuyerUI (or OpenStallOwnerUI for
-- the owner) by looking up the plotAnchorId attribute and resolving to a
-- player via the cached map of plotAnchorId -> userId Stall maintains.
local function buildPlotAnchorGrid()
    local PlotAnchors = require(ServerScriptService.Server.PlotAnchors)
    local layout = PlotAnchors.buildGridLayout()
    for _, entry in ipairs(layout) do
        local existing = Workspace:FindFirstChild(entry.tileId)
        if existing and existing:IsA("BasePart") then
            if existing:GetAttribute("plotAnchorId") == nil then
                existing:SetAttribute("plotAnchorId", entry.tileId)
            end
        else
            -- 16x16 stud square, slightly raised so it reads as a "plot"
            -- without colliding with player movement. CD-3: placeholder Part.
            local tile = makePart(
                entry.tileId,
                Vector3.new(16, 0.5, 16),
                entry.position + Vector3.new(0, 0.25, 0),
                Color3.fromRGB(110, 90, 70),
                Enum.Material.Slate
            )
            tile:SetAttribute("plotAnchorId", entry.tileId)
            tile.Parent = Workspace
            -- Stall stand on top — a small wooden post players can interact
            -- with via ProximityPrompt. Becomes the Owner's stall stand.
            -- Until a player claims this tile, the stand is unowned and the
            -- prompt is a no-op (Stall.start handler resolves nil userId silently).
            local stand = makePart(
                entry.tileId .. "_Stand",
                Vector3.new(2, 3, 2),
                entry.position + Vector3.new(0, 2, 0),
                Color3.fromRGB(140, 110, 80),
                Enum.Material.Wood
            )
            stand:SetAttribute("plotAnchorId", entry.tileId)  -- duplicated for prompt parent lookup
            stand.Parent = tile
            local prompt = Instance.new("ProximityPrompt")
            prompt.Name = "StallPrompt"
            prompt.ActionText = "Browse Stall"
            prompt.ObjectText = "Player Stall"
            prompt.HoldDuration = 0
            prompt.MaxActivationDistance = 10
            prompt.RequiresLineOfSight = false
            prompt.Parent = stand
        end
    end
end

**ProximityPrompt routing in Stall.start** (refresher from PLAN 02):

The Stall.start PromptTriggered handler receives every prompt trigger globally and must filter:
1. `prompt.Parent` should have attribute `featuredSlotIndex` (featured-row stand) OR `plotAnchorId` (plot tile or its child stand).
2. Resolve the seller userId:
   - For featured slots: read attribute `stallSellerUserId` (set by RentFeaturedSlot handler when a player rents). If nil, silent no-op (slot unrented).
   - For plot anchors: look up the tileId in Stall's in-memory `plotAnchorIdToUserId` cache (built from PlayerData scans at server start + maintained on each PlotAnchors.assign call). If nil, silent no-op (tile unclaimed).
3. If the triggering player's userId == sellerUserId → fire OpenStallOwnerUI to that player.
4. Else → fire OpenStallBuyerUI to that player with the seller's listings snapshot payload.

This logic should already be in PLAN 02's Stall.start handler. THIS plan (PLAN 03) is when it actually exercises (because the world Parts with these attributes now exist).

If PLAN 02's Stall.start did NOT implement this routing (per <interfaces> in PLAN 02 it should have), this plan adds it. Document the gap in the SUMMARY.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → server (ProximityPrompt trigger) | A player triggering a stall stand's prompt is server-validated by the PromptTriggered handler in Stall.start (already implemented in PLAN 02). The handler checks the prompt parent's attributes; if none match expected (`stallSellerUserId`, `featuredSlotIndex`, `plotAnchorId`), no remote fires. |
| client → server (DialogueAction → openStallOwnerUI / rentFeaturedSlot) | Dialogue.luau's handleAction (Dialogue.luau:120-134) re-validates: caller is in talk range, response exists in the tree, action.kind is in the known handler set. Plus the Stall.requestRentFeatured handler does its own D-15 validation matrix server-side. |
| Stall.start PromptTriggered handler → OpenStallBuyerUI / OpenStallOwnerUI | Server-side decision. Client cannot force the choice; the handler determines based on `stallSellerUserId` attribute vs triggering player's UserId. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-16 | T (Tampering) | ProximityPrompt parent attribute spoofing (a malicious client cannot set attributes — Roblox attributes are server-authoritative) | accept | Roblox doesn't allow client-side attribute writes on server-owned Parts. Players cannot spoof `stallSellerUserId` or `plotAnchorId`. Server is the only writer. |
| T-02-17 | E (Elevation of Privilege) | A player triggers another player's plot stand to fire OpenStallOwnerUI to themselves | mitigate | Stall.start PromptTriggered handler resolves the seller userId from the attribute, then compares to the triggering player's userId. Match → OpenStallOwnerUI; mismatch → OpenStallBuyerUI. The Owner UI can ONLY be opened by the actual owner; checked at the handler level not the client. |
| T-02-18 | I (Information Disclosure) | A buyer browsing a plot stand sees all of the seller's listings (including price + qty) | accept | This is the intended design — listings are public to anyone who walks up to the stall. Personal-only stalls are word-of-mouth discoverable per spec §15 (D-05); not listed in the global Plot Directory. |
| T-02-19 | T (Tampering) | Dialogue rentFeaturedSlot handler bypasses validation by firing the remote directly | mitigate | Stall.requestRentFeatured runs the full D-15 validation matrix server-side regardless of the entry path (dialogue vs direct remote). The dialogue handler is a thin pass-through. |
| T-02-20 | D (Denial of Service) | World build adds 64 plot tiles + 12 featured stands + 1 NPC = 77 new server Parts on every server start | accept | One-time cost at server boot. Each tile is a 16x16 Part (small relative to terrain). Per-frame cost: zero (Anchored, no scripts attached at the Part level — only ProximityPrompts which fire only on player input). |
| T-02-21 | T (Tampering) | Plot anchor stamp race — two players walking to two unclaimed tiles simultaneously could in theory both try to claim the same lowest-free tile | mitigate | Stall.luau's PromptTriggered handler runs server-side serially (one OnClientEvent at a time per handler). Atomicity within a single PlayerData.update call. The "scan claimedSet → assign tile" sequence runs in one handler invocation, so two simultaneous prompts produce two sequential assignments to two different tiles (lowest-free vs second-lowest-free). No race. |
</threat_model>
